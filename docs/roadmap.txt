lit - Roadmap
=============

Hard constraints that apply to every phase:
- Zero-latency live tweaks — any parameter change must reach the math with zero perceptible delay
- Native processing speed — no interpreted layers, no shader recompilation on parameter change, no GC pauses, no allocation on the hot path
- Parameter values live in shared memory / uniform buffers accessible to both CPU and GPU; CPU writes, GPU reads next frame
- This shapes every architectural choice. If a feature can't be made zero-latency, it doesn't belong in the live path.

Bake constraint (settled):
- Topology is baked at build-time. Runtime only varies parameters.
- Build-time can be slow (compilation takes seconds to minutes). Live-time is just uniform writes.
- Topology mutation never happens live. The substrate compiles a node graph into a runtime artifact once; that artifact runs unchanged during performance.
- Mirrors VST plugins, GPU shaders, FPGA synthesis.
- Note: bake is a spectrum, not binary. Start interpreted (simplest), measure, add bake only as much as needed.

Program split (architecture, three binaries):
- The system is four things (three binaries + external):
  1. **Builder** — substrate editor + visual field preview. Dev-only. Compiles and bakes Intentions.
  2. **Operation** — runtime Intentions + busking surface + MIDI/OSC + DMX output. Lives at the show. Lean.
  3. **Visualizer** — spatial field preview + receives baked shaders from Builder. Runs on beefy hardware with a real GPU.
  4. **(External) Capture** — real fixture viz, GDTF-based. Used as-is.
- All communicate over OSC (standard show-control protocol).
- Three binaries = maximum flexibility. Each deploys independently based on the show's needs.

Compute target (revised — CPU is default):
- CPU evaluation is the DEFAULT. MA2 onPC does 4-8 universes of pixel mapping on a single CPU core with limited RAM. Same workload class.
- GPU compute is OPTIONAL for scale: pixel walls, 3D lighting rigs with hundreds of fixtures, complex spatial math.
- Operation PC: a recent i3 + 16GB RAM is enough for typical busking shows. No discrete GPU required for most cases.
- The substrate runs interpreted (CPU) or compiled (GPU). Per-Intention decision based on graph size.
- This means most shows run on CPU only. Big shows get the GPU acceleration.

Bake when it matters (scale-dependent, revised):
- Small rigs: interpreted evaluation, no bake needed.
- Large rigs (pixel walls, 3D lighting rigs with hundreds of fixtures): bake starts to matter.
- Per-Intention decision, not all-or-nothing.
- Pixel walls and 3D lighting rigs (3D sphere or cube) are the trigger cases for full bake.

This is the build order for the system. Each phase ends in something you can actually use, not just code that compiles.

Each phase should also mark which program(s) it touches (Builder, Operation, Visualizer).

Phase 0 – Validate the toolchain (DONE)
- Status: complete
- What got built: GPU compute to SSBO to CPU readback to ImGui visualization loop
- What it proves: the foundation works (window, GL context, GPU math, data roundtrip)
- The UI prototypes (`lit_view`, `lit_playground`) have been archived. They served their purpose.

Phase 1 – Shader‑field research (current)
- Exit criterion: a self‑contained C++ program that runs a GLSL compute shader
  producing a moving scalar/color field, samples it at virtual fixture positions,
  and displays the result in a window. No node graph, no UI toolkit—just raw
  OpenGL + SDL3.
- Why: prove that the “Fixtures as Pixels” concept works at all, before we build
  tooling around it. This is the foundation for everything else.
- What gets built:
  - Single‑file experiment: SDL window + compute shader that generates a sine wave
    field over a 2D grid; read back and display with glDrawPixels or a full‑screen quad.
  - Extension: define a set of fixture positions (hardcoded), sample the field at
    those positions, display the sampled colors alongside the field.
  - Extension: re‑implement the diagonal red sweep entirely in a shader (no substrate)
    to confirm that complex spatial effects can live on the GPU.
- Deliverable: a `src/experiments/` directory with numbered, standalone programs.
- No UI libraries, no node graph runtime. Pure research.

Phase 2 – Substrate‑to‑shader compilation
- Exit criterion: a substrate `Graph` can be compiled into a GLSL compute shader
  that produces identical output to the CPU‑side evaluation for a simple effect.
- What gets built:
  - `GraphCompiler` class that traverses a baked graph and emits GLSL function
    calls for each node type.
  - Tests that compare CPU and GPU output for small graphs (Phase→Sine→Output).
- This is the “bake” step made concrete. It turns the substrate from an interpreted
  CPU library into a GPU code generator.

Phase 3 – Busking surface (Operation)
- Exit criterion: a touch‑friendly executable that loads a compiled shader,
  displays a 3D field preview, and lets you tweak parameters via faders and buttons.
- UI: retained‑mode, touch‑first (e.g., Qt Quick or Slint), not immediate‑mode.
- Executor grid, tap‑tempo, master faders, and symmetry controls.
- This is the live performance binary.

Phase 4 – DMX/Art‑Net/sACN output
- Exit criterion: real fixtures respond to the busking surface.
- Protocol output layer, per‑universe scheduling, fixture profile mapping.

Phase 5 – GDTF visualizer and polish
- Real fixture models in a 3D scene.
- Library management, MIDI/OSC input, audio FFT.
