# Experiment 04 – Visual Fixture Overlay

**Date:** 2026-07-18  
**Goal:** Prove that the structured GPU output (`GPUFixureOutput`) can be
rendered as coloured fixture points on top of the background field, with
dimmer mapped to brightness. This validates the visualisation pipeline:
compute → SSBO → CPU readback → point rendering.

## Result

Success. The window shows a moving grayscale sine wave (background field)
with ten red squares at the V‑shaped Bar L/R probe positions. The squares
pulse in brightness independently of the background field, confirming that
the dimmer channel from the SSBO is correctly displayed. The HSL data is
present in the SSBO but intentionally not used for point colour (kept pure
red for unambiguous verification).

## What it proved

- The `GPUFixureOutput` struct travels correctly from the compute shader
  (via SSBO) to the CPU, with correct alignment and values.
- The CPU can read the dimmer channel and use it to modulate the brightness
  of rendered points.
- Point rendering (VAO/VBO + tiny vertex/fragment shaders) works alongside
  the compute‑and‑blit pipeline without interfering with ImGui‑style state.
- The background field and the fixture overlay are independent; this
  isolation made it possible to verify each piece separately.

## Design decisions

### Why red squares instead of HSL‑to‑RGB conversion?

Earlier attempts used the HSL fields to colour the points, but the results
were confusing (rainbow hues, white at high lightness). To prove that
dimmer→brightness mapping works, we used a fixed red colour scaled by
dimmer. The HSL data is still generated by the shader (hue=0, full sat,
lightness=dimmer) and stored in the SSBO for future profile mapping, but
the visualisation ignores it. This separation keeps the test clean.

### Why the background field is not sampled by the fixtures

The fixture dimmer uses its own formula (`sin(u_time + u * π) * 0.5 + 0.5`),
not a sample of the background field. This was intentional: if the points
were sampled from the field, a bug could cause them to appear correct by
accident. Independent behaviour makes it obvious that the SSBO data path
works correctly. Experiment 05 will connect them.

### Texture format change: R32F → RGBA32F

The background field texture was changed from single‑channel (`R32F`) to
four‑channel (`RGBA32F`) because `glBlitFramebuffer` only writes to the
red channel when the source is `R32F`, producing a red‑tinted image.
Writing the same intensity to all four channels produces a proper grayscale
display.

## Open issues

- Point size is fixed at 12 pixels; no scaling with window size.
- Fixture labels are printed to the terminal, not drawn on screen (would
  require a text rendering library or bitmap font).
- The window is not resizable (points don't reposition correctly after
  resize — to be fixed in a future experiment).

## Next steps

Experiment 05 will connect the background field and the fixture output:
the fixture dimmer will be a true sample of the field at its UV position,
closing the “Fixtures as Pixels” loop. After that, the diagonal red sweep
can be implemented as a shader and visualised with the same pipeline.
